# job-manage 概述

## job-manage 是什么

`job-manage` 是节点管理体系中的任务管理服务，负责面向被 `nodemanage` 纳管的节点执行批量操作，例如脚本分发、命令执行、文件传输、任务编排、执行状态跟踪和结果回收。

它不负责节点注册，也不负责节点在线状态计算。它依赖 `nodemanage` 获取节点清单和节点状态，并通过 `rsagent` 完成节点侧的实际执行。

一句话定位：

> `job-manage` 是面向纳管节点的远程任务编排与执行管理服务。

## job-manage 解决什么问题

### 1. 节点操作需要从人工命令升级为平台任务

没有 job-manage 时，对多台节点执行脚本、命令或文件分发通常依赖人工 SSH、临时脚本或批量工具。这种方式难以审计、难以追踪进度，也不方便失败重试。

`job-manage` 将这些操作抽象成平台任务，让用户可以提交任务、选择目标节点、查看执行进度、获取执行结果。

### 2. 批量节点操作需要统一调度和状态管理

同一个任务可能需要在几十台或几百台节点上执行，每个节点可能有不同的执行状态，例如等待中、执行中、成功、失败、超时。

`job-manage` 负责维护任务级和节点级状态，支持并发控制、失败重试、超时处理和结果聚合。

### 3. 远程执行需要和节点状态联动

任务执行前需要判断目标节点是否在线，是否最近有心跳，是否具备执行条件。这个判断不应该由用户手工完成。

`job-manage` 可以基于 `nodemanage` 提供的节点状态，在任务下发前过滤离线节点，或将不可执行节点标记为跳过/失败。

### 4. 执行能力需要沉淀为可复用产品能力

脚本分发、命令执行和文件传输是运维场景中的基础能力。将它们沉淀到 job-manage 后，可以支撑后续更多产品场景，例如批量巡检、配置变更、日志采集、故障修复和自动化运维。

## 与其他组件的关系

- nodemanage：提供被纳管节点的基础信息、节点状态和节点选择范围。
- `rsagent`：在节点侧接收任务、执行脚本/命令/文件操作，并回传结果。
- job-manage：负责任务创建、调度、状态跟踪、结果聚合和审计记录。

## 第一阶段目标

job-manage 第一阶段优先支持最小可用的远程执行闭环：

1. 创建任务：支持脚本执行与命令执行两类基础任务。
2. 选择节点：从 `nodemanage` 纳管节点中选择目标节点。
3. 执行前检查：通过 `nodemanage` 提供的节点状态判断目标节点是否可执行。
4. 下发任务：通过 rsagent 通信通道将任务发送到节点。
5. 跟踪状态：记录任务整体状态和每个节点的执行状态。
6. 回收结果：收集完整 stdout、完整 stderr、退出码、错误信息和执行耗时。

第一阶段完成后，平台将具备从“管理节点”进一步升级为“操作节点”的基础能力。
