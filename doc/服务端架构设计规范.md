# 服务端架构设计规范

## 1. 文档目的

本文档沉淀 rsde 仓库内服务端设计的通用原则，用于指导新模块设计、存量模块演进，以及跨模块能力的归档与复用。

适用范围：

- `apiserver`
- 独立领域 crate（如 `datalink-engine`）
- 后续新增的服务端模块
- `common/*` 下的共享能力设计

本文档作为**仓库级原则**使用，优先约束新增设计；存量实现按具体场景渐进靠拢，不要求为满足规范而做脱离业务价值的大规模重构。

---

## 2. 总体原则

### 2.1 优先做资源声明式接口，而不是事件式接口

服务端在设计对内或对外 API 时，默认优先采用**资源声明式**的接口模型，让调用方表达“我希望资源处于什么状态”，而不是要求调用方按事件顺序驱动服务端内部流程。

优先推荐的资源型操作包括：

- `get`：查询单个资源
- `list`：查询资源集合
- `apply`：声明式创建或更新资源期望状态
- `delete`：删除资源或进入删除态

必要时可以补充：

- `patch`：局部修改资源
- `status`：资源状态变更或状态子资源更新

这类接口的目标是：

- 保持 API 抽象稳定，避免把服务端内部执行步骤暴露给客户端
- 让调用方只关注资源及其期望状态，而不是关注内部编排顺序
- 让幂等、重试、对账、状态收敛更容易实现
- 降低跨模块协作时的心智负担，提升维护性

### 2.2 事件模型不是禁用，而是默认不优先

事件模型并非禁止使用，但默认不作为首选。

只有在以下场景中，事件式接口更自然时，才采用事件模型：

- 动作天然一次性且不可抽象为稳定资源状态
- 业务语义核心是“触发某次动作”，而不是“维护某个资源当前状态”
- 事件本身需要保留完整时序语义、审计语义或异步流程语义
- 资源抽象会明显扭曲业务概念，导致接口更难理解

即使采用事件模型，也应先回答：

1. 是否真的不存在更合适的资源抽象？
2. 事件触发后是否会形成一个可查询、可跟踪的资源对象？
3. 是否需要为事件结果补一个稳定的查询对象或状态对象？

如果答案倾向于“可以抽象成资源”，则应回到资源声明式设计。

### 2.3 服务端负责收敛状态，客户端负责表达意图

调用方应尽量表达意图和目标状态，服务端负责：

- 校验输入
- 生成或维护资源标识
- 对齐实际状态与期望状态
- 管理内部执行顺序
- 保证幂等、重试和一致性边界

不推荐把“先调 A，再调 B，再调 C”这类内部操作顺序强加给客户端。客户端如果必须了解大量内部步骤，通常意味着服务边界或资源抽象还不够稳定。

## 3. 资源声明式接口设计约定

### 3.1 优先围绕资源建模

设计接口前，先明确以下问题：

- 这个服务真正管理的资源是什么？
- 资源的唯一标识是什么？
- 资源有哪些稳定字段、状态字段、只读字段？
- 调用方期望声明的是“目标状态”还是“执行动作”？

如果一个能力可以被描述为“某类对象的创建、查询、对齐、删除、列举”，就应优先建模为资源。

### 3.2 `apply` 优先于碎片化事件接口

当客户端需要维护一个资源的整体期望配置时，优先提供 `apply` 类接口，而不是拆成多个强顺序事件接口。

推荐：

- `ApplyXxx` / `PUT /xxx:apply`

不推荐默认拆成：

- `CreateXxx`
- `BindYyy`
- `EnableZzz`
- `SyncAaa`

除非这些动作本身分别具有独立稳定的业务语义，否则应由服务端内部完成编排和收敛。

### 3.3 幂等是默认要求

资源声明式接口应默认按幂等接口设计，尤其是 `apply`、`delete`、`status` 类操作。

需要明确：

- 幂等键或资源唯一键是什么
- 重复请求的返回语义是什么
- 服务端如何处理“资源已存在但期望状态一致/不一致”

### 3.4 资源标识优先稳定主键，不优先业务别名

跨组件引用时，优先使用稳定 ID，而不是名称、展示字段或临时别名。

例如：

- 组件间稳定引用优先使用 `data_link_id`
- `result_table_name` 可以作为管理检索入口，但不应默认承担主要跨组件主键角色

### 3.5 状态流转要显式约束

如果资源有生命周期状态，必须明确：

- 状态集合
- 合法流转路径
- 禁止流转路径
- 状态说明字段的约束

不要把状态语义散落在调用方约定、注释或隐式判断里。

### 3.6 涉及两个及以上组件的对接协议时，优先考虑显式版本号

当一个协议需要由两个及以上组件共同对接，并且协议后续可能独立演进时，默认应考虑增加**显式版本号**，避免协议演进只能依赖口头约定或代码猜测。

版本号可以放在以下位置：

- API 路径中，例如 `/api/datalink/v1/...`
- 协议字段中，例如 `version`、`schema_version`、`protocol_version`
- 握手、注册、配置拉取等协议对象中

这样做的价值包括：

- 便于兼容演进和灰度发布
- 便于调用方和服务端做兼容性判断
- 便于排障、回滚和协议变更审计
- 避免多个组件各自演进后出现“看起来字段差不多，但语义已经漂移”的问题

默认建议是：

- 只要协议会被两个及以上组件长期依赖，就优先考虑从第一版开始显式标记 `v1`
- 版本号一旦对外暴露，就应保持语义稳定，不要随意改写同一版本的含义
- 重大不兼容变更通过新版本表达，而不是静默覆盖旧行为

如果某个调用只是模块内私有协作、双方不会独立演进，或者协议寿命明确非常短，则可以不强求单独版本化；但在设计文档中应明确说明为什么不需要版本号。

## 4. `common` 共享能力设计规范

### 4.1 `common` 的定位

`common/*` 用于统一维护仓库内**基础、通用、跨模块复用**的共享能力，而不是承载某个具体业务模块尚未想清楚的临时逻辑。

当前仓库中，`common/config` 已经体现出较清晰的共享定位：

- 统一配置模型
- 统一配置加载与校验入口
- 多个服务共同依赖
- 不反向依赖业务模块

这类能力适合沉淀在 `common`。

### 4.2 进入 `common` 的准入原则

默认采用以下准入标准：

1. **领域无关或弱领域相关**：不是某个单一业务模块私有语义的实现细节
2. **跨模块复用**：至少明确有多个模块会长期使用，而不是“可能以后会复用”
3. **语义稳定**：接口和职责已经相对清晰，不处于频繁试错阶段
4. **边界清晰**：能说清它做什么、不做什么、依赖什么
5. **不倒灌业务依赖**：不能让 `common` 反向依赖具体业务 crate

可以简单理解为：

> `common` 不是“代码复用回收站”，而是“稳定共性能力层”。

### 4.3 不应进入 `common` 的内容

以下内容默认不应直接放入 `common`：

- 单一业务模块专属的领域模型
- 仍在快速演化、边界未稳定的能力
- 为了“避免复制几行代码”而过早抽取的工具函数
- 带有明显业务前缀、业务状态机、业务流程编排的逻辑
- 依赖具体服务实现细节的适配层

如果某段代码当前只服务于一个模块，即使看起来“以后可能复用”，也优先保留在该模块内部，待语义稳定且出现真实复用需求后再上收。

### 4.4 推荐的上收路径

当某项能力需要进入 `common` 时，推荐按以下顺序判断：

1. 先在所属模块内完成抽象和验证
2. 出现明确的第二个使用方
3. 收敛公共接口、依赖边界和命名
4. 再迁移到 `common/*`
5. 同步补充 README 或设计文档，说明职责和依赖关系

这比一开始就把未成熟逻辑塞进 `common` 更可维护。

## 5. 推荐的服务边界划分

### 5.1 独立领域能力优先做独立 crate

当一个能力拥有独立的领域模型、服务规则和仓储抽象，并且预期会被多个组件依赖时，优先设计成独立 crate，由 `apiserver` 负责 HTTP 路由集成和进程级装配。

推荐模式：

> 独立 crate 承载领域模型与核心逻辑，`apiserver` 负责路由、状态注入和装配。

### 5.2 route 层不承载核心业务规则

路由层主要负责：

- 协议解析
- 参数绑定
- 调用 service
- 响应映射

不应在 route 层堆放：

- 核心业务校验
- 状态机规则
- 仓储拼装细节
- 跨资源编排逻辑

这类逻辑应放在领域 service 或独立模块内。

## 6. 文档与评审要求

当新增服务端模块或新增重要 API 时，设计文档至少应说明：

- 管理的资源是什么
- 为什么采用资源模型或为什么不采用
- 资源主键、检索键、状态字段分别是什么
- 是否支持 `get / list / apply / delete` 中的哪些操作
- 幂等语义和状态流转如何定义
- 如果涉及两个及以上组件的对接协议，是否需要显式版本号，以及版本放在哪里
- 是否有能力应沉淀到 `common`，以及为什么

如果选择事件模型，应在设计文档中明确说明不采用资源声明式模型的原因，而不是默认跳过这一层分析。

## 7. 正反例

### 7.1 推荐示例：DataLink 类接口

类似以下能力，适合资源声明式建模：

- `ApplyDataLink`
- `GetDataLink`
- `ListDataLinks`
- `SetDataLinkStatus`

原因在于：

- 核心对象是稳定资源 `DataLink`
- 客户端表达的是链路期望配置和状态
- 服务端负责 create/update/reconcile
- 资源主键、状态、查询入口都可以清晰定义

### 7.2 不推荐示例：暴露内部编排步骤给客户端

如果一个能力本质上只是“维护一条资源”，却把接口设计成：

- `CreateDatasource`
- `CreateResultTable`
- `BindDatasourceToResultTable`
- `EnablePipeline`
- `FinalizeLink`

则通常意味着：

- 服务端内部边界泄漏到了 API 层
- 客户端必须理解服务端内部顺序
- 重试、补偿、幂等和演进成本更高

除非这些动作本身就是独立资源，否则不推荐这样设计。

---

## 8. 落地建议

1. 新增服务端能力时，先问“资源是什么”，再问“接口怎么命名”。
2. 新增共享能力时，先放模块内抽象，稳定后再考虑进入 `common`。
3. 设计评审时，默认追问是否可以从事件式接口收敛为资源声明式接口。
4. 对存量代码，不为追求形式统一而强行重构；只在新增需求或修改路径上渐进对齐。

这份规范的目标不是限制实现方式，而是让仓库内的服务端设计长期保持：

- 抽象稳定
- 接口简洁
- 边界清晰
- 维护成本可控
