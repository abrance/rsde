
### 1 事项工程目录结构

每个事项建立独立目录，命名规范：`items/{事项标题简写}/`

```
items/{title}/
├── item_info.md              # 事项入口与资料信息
├── research.md               # 研究产物
└── docs/
    ├── architecture.md       # 架构与关键数据流
    ├── conventions.md        # 编码与协作规范
    └── plans/
        ├── active/           # 当前生效的计划
        │   └── plan.md
        └── completed/        # 已完成归档的计划
```

### 1.2 Step 1 — 研究（Research）

**产物**：`research.md`

必须回答以下问题（涉及代码层面）：

| # | 必答问题 | 说明 |
|---|---------|------|
| 1 | 现有代码逻辑怎么工作？ | 梳理当前实现的核心流程、入口、关键函数 |
| 2 | 这次改动涉及哪些模块和调用链？ | 列出受影响的文件、模块、上下游依赖 |
| 3 | 可复用什么，不该动什么？ | 识别可复用组件和不可触碰的稳定区域 |
| 4 | 风险点和证据是什么？ | 列出潜在风险及支撑判断的代码/日志/文档证据 |

### 1.3 Step 2 — 规划（Plan）

**产物**：`docs/plans/active/plan.md`

必须包含：

| # | 必含内容 | 说明 |
|---|---------|------|
| 1 | 实现方案与边界 | 做什么、不做什么、方案的适用范围 |
| 2 | 修改/新增文件路径 | 涉及的所有文件，逐一列出 |
| 3 | 关键代码片段 | 核心逻辑的伪代码或代码草案 |
| 4 | 方案权衡与选择理由 | 为什么选这个方案而非其他备选 |
| 5 | 可勾选任务清单 | `- [ ]` 格式的执行步骤，含验证命令 |

### 1.4 Step 3 — 批注请求（Review Request）

管理者在 `plan.md` 中逐条批注，执行者只修计划不写代码，直到计划定稿。

```
循环流程：
  管理者批注 → 执行者修改计划 → 管理者再审 → ... → 定稿 ✅

定稿标志：管理者在 plan.md 顶部标注「✅ APPROVED」
```

### 1.5 Step 4 — 实施（Execute）

按定稿计划**顺序**执行，完成一项勾一项。

**推荐 TDD 流程**：先写测试，再改实现。

```
执行规则：
1. 严格按照 plan.md 中的任务清单顺序执行
2. 每完成一个步骤，立即在 plan.md 中勾选 [x]
3. 如遇计划外问题，先回到 plan.md 补充步骤，再继续执行
4. 不得跳步、合并步骤或偏离计划
```

### 1.6 Step 5 — 验收与回灌（Verify + Sync）

| # | 验收动作 | 说明 |
|---|---------|------|
| 1 | 跑验证命令 | 执行 plan.md 中定义的所有验证命令，确保全部通过 |
| 2 | 更新 `item_info.md` | 回灌最终状态：完成时间、产出物链接、变更摘要 |
| 3 | 更新 `docs/architecture.md` | 如架构有变，同步更新架构文档 |
| 4 | 归档计划 | 将 `docs/plans/active/plan.md` 移至 `docs/plans/completed/` |
| 5 | 更新 Vikunja 任务状态 | 标记完成、更新进度百分比 |

### 1.7 采访驱动启动

在启动事项的 Step 1（研究）之前，通过**结构化采访**了解管理者/需求方的想法和意图：

```
采访规则：
1. 先结构后细节，一步步递进
2. 除非是并行问题，否则一个问题一个问题地问
3. 当回答不清楚时，追问澄清
4. 整体不超过 10 个问题
5. 达到 95% 信心理解真实需求和目标时，结束采访
```
