# 命令行工具整理

## 1. 文档目的

本文档用于整理 rsde 仓库在日常编码、构建、联调、发布、排障过程中依赖的命令行工具与运行环境，避免开发时只关注单个语言工具链，而遗漏实际工作中必需的外围工具。

本文档分为两类：

1. **本地命令行工具链**：直接安装在开发机上，用于编码、构建、测试、提交、发布。
2. **集群内调试/发布依赖**：不一定以本地二进制存在，但在发布和调试阶段需要通过 Kubernetes 命名空间内的服务配合使用。

---

## 2. 当前已确认的本地工具链

根据当前环境与仓库文档，已确认本地存在以下工具：

### 2.1 Rust 工具链

- `cargo`
- `rustc`
- `rustfmt`
- `cargo clippy`

已知路径：

- `~/.cargo/bin/cargo`

仓库中的主要用途：

- `cargo build --release`
- `cargo test --workspace`
- `cargo fmt -- --check`
- `cargo fmt`
- `cargo clippy --workspace --all-features`
- `cargo run -p apiserver`

说明：

- Rust 是本仓库的主工具链。
- `AGENTS.md`、根 `Makefile`、CI workflow 都明确依赖 Rust 工具链。
- CI 中还显式安装了 `rustfmt` 和 `clippy`，说明这两者不是可选增强，而是标准开发链路的一部分。

### 2.2 Node.js / npm / npx 工具链

- `node`
- `npm`
- `npx`

已知路径：

- `~/.nvm/versions/node/v24.11.1/bin/node`

当前环境版本：

- Node.js `v24.11.1`
- npm `11.6.2`
- npx `11.6.2`

仓库中的主要用途：

- `npm install`
- `npm run dev`
- `npm run build`
- `npm run lint`

说明：

- 前端位于 `webserver/frontend`，技术栈为 React + TypeScript + Vite。
- 根 README、`AGENTS.md`、CI workflow 都要求先构建 frontend，再运行 `apiserver` 或打包 Docker 镜像。
- 因此前端工具链虽然不是仓库主语言，但属于日常开发的刚需。

### 2.3 TypeScript 工具链

- `tsc`（当前通过 `npx tsc` 使用）

当前环境版本：

- TypeScript `6.0.3`（来自 `npx tsc -v`）

仓库中的主要用途：

- `webserver/frontend/package.json` 中的 `build` 脚本为：`tsc && vite build`

说明：

- TypeScript 实际上已经是前端构建链路的一部分，不应只笼统地记为“Node.js 工具链”。
- 若前端页面或接口类型调整，`tsc` 是第一层静态校验工具。

### 2.4 Git / GitHub CLI

- `git`
- `gh`

当前环境：

- `git` 已存在
- `gh` 已存在

说明：

- `git` 是基础开发工具，不单独解释。
- `gh` 适合用于 PR、Release、GitHub 相关协作流程；当前仓库已有 GitHub Actions 和 Release workflow，因此 `gh` 属于建议保留的协作工具。

### 2.5 Make

- `make`

仓库中的主要用途：

- `make build`
- `make test`
- `make fmt`
- `make fmt-check`
- `make clippy`
- `make run-apiserver`

说明：

- `Makefile` 已将常见命令做了统一封装。
- 对于团队协作来说，`make` 是“统一入口工具”，虽然它不是语言工具链，但仍属于编码时的常用 CLI。

### 2.6 Docker

- `docker`

当前环境：

- 本地已存在 `docker`

仓库中的主要用途：

- `make docker-build`
- `make docker-push`
- CI 中的 Docker image build / push

说明：

- Docker 不仅用于最终发布，也用于本地验证镜像构建是否正常。
- 对本仓库来说，Docker 已经是开发到发布链路之间的标准环节。

### 2.7 Kubernetes / Helm 工具链

- `kubectl`
- `helm`

当前环境：

- 本地已存在 `kubectl`
- 本地已存在 `helm`

仓库中的主要用途：

- 查看集群资源、日志、事件
- 部署/升级 `helm/rsde`
- 回滚 Helm release
- 做 chart lint 和 template dry-run

说明：

- `AGENTS.md`、Helm README、CI workflow 都明确包含 `kubectl` / `helm`。
- 这两者对本仓库不是运维专用边缘工具，而是本地联调和发布时的标准工具链。

### 2.8 jq

- `jq`

说明：

- 适合在调试 JSON API、查看 Kubernetes JSON 输出、处理配置响应时使用。
- 虽然当前仓库文档没有把 `jq` 当成硬性要求，但从实际工作流看，它属于非常常用的辅助工具，建议纳入统一整理。

### 2.9 bun

- `bun`

说明：

- 当前仓库前端主链路仍然基于 `npm`，CI 也是 `setup-node + npm cache`。
- 因此 `bun` 当前更适合作为**本地补充工具**，不是仓库标准前端包管理器。
- 建议在文档中明确：可安装保留，但不要默认把项目脚本或锁文件切到 bun 生态，除非后续团队明确迁移。

## 3. 从仓库证据看，编码阶段需要关注的工具链范围

如果只从“写代码”狭义理解，容易只想到 `cargo` 和 `node`。但结合当前仓库实际，编码阶段需要关注的工具链至少包括以下层次：

### 3.1 代码编译与静态校验

- Rust：`cargo`、`rustfmt`、`clippy`
- Frontend：`node`、`npm`、`npx`、`tsc`

### 3.2 本地统一命令入口

- `make`

### 3.3 仓库协作与版本管理

- `git`
- `gh`

### 3.4 镜像与部署验证

- `docker`
- `kubectl`
- `helm`

### 3.5 调试辅助

- `jq`

### 3.6 可选补充工具

- `bun`

结论是：

> 对 rsde 来说，“编码时需要的工具链”不只是 Rust 和 Node.js，还应包含格式化、lint、前端构建、Make 统一入口、容器构建、K8s 部署验证、JSON 调试等整套 CLI 能力。

### 4.2 为什么要单独整理

因为它们承担的是：

- 服务联调目标环境
- 数据写入与查询验证环境
- 配置、消息、缓存、索引、时序数据调试环境
- 发布后问题排查环境

如果只记录“本地有 cargo 和 node”，但不记录 `bkbase-test` 中有哪些依赖，团队在实际调试时仍然会缺少关键上下文。

### 4.3 建议的归类方式

建议在后续持续补充时，把 `bkbase-test` 里的依赖按职责归类：

- **关系型存储**：MySQL
- **缓存 / KV**：Redis
- **日志 / 检索**：Elasticsearch
- **消息系统**：Kafka
- **时序指标**：VictoriaMetrics

后续如果需要，还可以继续补充：

- 各服务在 `bkbase-test` 中的资源名
- 对应 Service 名称
- 访问方式（集群内访问 / port-forward / ingress）
- 常用排障命令

补充说明：

- `*.svc.cluster.local` 是 Kubernetes 集群内服务地址。
- 当前本地环境已将 K8s CoreDNS 暴露到本机，因此本地终端可以直接解析并访问这些 `*.svc.cluster.local` 地址。
- 也就是说，在当前环境下，这些 `bkbase-test` Service 地址可以直接作为本地联调地址使用，而不需要额外做 `kubectl port-forward`。
- 如果未来更换到未暴露 CoreDNS 的环境，再回退到 `kubectl port-forward`、跳板机、VPN 或在 Pod 内执行调试命令等方式。

## 6. 后续建议

1. 后续如果新增新的语言或构建链路，例如 Python 脚本、MCP 服务、数据库迁移工具，应继续补到本文档。
2. 如果团队准备把某个工具升级为“标准链路”，例如把 `bun` 从补充工具变为正式包管理器，应同步修改本文档与 CI。
3. 如果 `bkbase-test` 命名空间中的依赖有固定访问方式，建议再单独补一篇“联调环境访问说明”。
4. 本文档的目标不是追求罗列完整，而是保证开发、调试、发布时不会遗漏关键工具与环境。

---

## 7. 当前最小结论

目前不应再把 rsde 的开发环境简单理解为：

- `cargo`
- `node`

更准确的最小集合应该是：

- Rust：`cargo` / `rustfmt` / `clippy`
- Frontend：`node` / `npm` / `npx` / `tsc`
- Repo：`git` / `gh`
- 统一入口：`make`
- 构建与部署：`docker` / `kubectl` / `helm`
- 调试：`jq`
- 可选补充：`bun`
- 环境依赖：`bkbase-test` 中的 MySQL / Redis / ES / Kafka / VM

这套整理方式更接近当前仓库的真实工作流。
